## 方向

用户选择融合方向 `1 高效运营型` + `2 社区生鲜型`：

- PC 后台：以高效运营为主，强调订单处理、筛选、批量发货、库存预警和权限边界。
- 小程序：以社区生鲜为主，强调套餐次数状态、真实菜品图、默认地址和确认下单。

## 设计说明

会员用户和后台用户在原型中保持分离：

- `会员用户` 用于管理 C 端用户资料、套餐、地址和订单。
- `后台用户与权限` 用于管理系统登录账号、角色和操作权限。
- `后台登录` 面向运营管理员和超级管理员，使用后台用户账号登录，并在登录后按角色展示菜单和操作权限。
- `后台两级导航` 当前采用工作台 + 订单管理、会员管理、套餐管理、任务管理、系统管理分组，二级菜单承载具体页面；一级分组必须配置独立图标，菜单支持展开 220px 和折叠 72px 两种状态，折叠后只显示一级图标，有二级菜单的入口在悬停时弹出二级菜单列表。
- `门店概念` 当前先只作为数据字段和权限边界保留，不在后台导航露出“门店管理 / 加盟门店 / 加盟商”，小程序也隐藏门店切换和当前门店卡片；后续若重新启用多门店/加盟店，再恢复菜单入口和小程序切换。
- `配送范围` 已补充 Figma 规格节点 `15 PC 后台 / 门店字段与配送范围`，数据层按门店预留配送省份、配送城市字段；空配置表示不限制，地址保存、默认地址切换、提交/修改预订都会走同一套校验。
- `侧栏收缩按钮` 固定在侧栏右上边界，使用“面板 + 箭头”图标按钮，不放在菜单列表内部。
- `二级菜单收缩` 已补充系统管理示例：一级分组标题带 chevron 图标，点击可收起后台用户、角色管理、菜单管理、操作日志、系统设置等二级菜单；行内不显示“展开/收起”文字，当前路由所在分组默认保持展开，手动状态可记忆。
- `后台用户区` 顶部右侧不只显示用户名，改为头像、姓名、角色、数据范围和下拉菜单组合；下拉菜单预留个人资料、切换角色、操作日志、退出登录。
- `后台字号` 已提高可读性：菜单一级分组标题使用 15px，关键一级入口使用 16px，二级项使用 15px，表格和正文基础字号不低于 13px；小程序避免 11px 以下文字。
- `后台工作弹窗` 用于详情、编辑、新建，支持拖拽、全屏和伸缩，替代右侧详情抽屉和独立详情页；系统设置这类单配置页也只在摘要页展示当前配置，点击编辑后进入同一套弹窗；右上角全屏控制使用四角最大化图标，不使用文字按钮。
- `小程序登录` 面向会员用户，参考星享俱乐部登录页的“品牌居中 + 大留白 + 单主按钮 + 底部协议”结构，使用蔬菜照片作为圆形会员视觉，只保留手机号授权登录；无套餐引导不放在登录页。
- `小程序导航` 底部 Tab 固定为：首页、我的；两个 Tab 必须配置默认和选中图标，不设置单独的预订、订单、套餐菜单。首页就是预订入口，订单和套餐从“我的”页入口进入。
- `小程序顶部` 首页、我的、登录、订单、套餐、地址管理主流程页面使用自定义导航，让页面背景延伸到状态栏和微信胶囊区域；页面正文不再显示原生白色标题栏里的“首页/我的/账号登录/订单/套餐/地址管理”。运行代码统一渲染 `MiniCustomTop` 安全区组件，并按微信 `statusBarHeight` 与胶囊尺寸动态留白，避免只配置 `navigationStyle: custom` 但页面顶部没有实际结构，或原生标题与页面内标题重复。
- `小程序预订` 已按 2026-06-18/19 反馈重做并同步到 Figma `03 小程序 / 首页（有套餐）`：首页不再展示菜品类型分类，改为可配置菜品网格卡片，默认每行 3 个；每张卡片使用菜品图片、名称、当前斤数和加减按钮，代码按容器响应式均分列宽，在 390px Figma 画板下等效为 110px 卡片，缺少接口图片时用本地蔬菜图兜底；页面正文去掉冗余“首页”大标题；配送地址从顶部卡片移到底部提交浮层，右侧按钮按状态显示“新增地址”或“切换”。
- `首页套餐展示` 套餐卡只展示套餐总次数、已用次数、剩余次数和右上角截单时间，数字优先，不再展示右侧装饰蔬菜图，也不重复底部已选重量。
- `首页今日已预订态` 用户当天已提交预订后，首页重新进入时直接回填今日菜品数量，底部主按钮从“提交预订”切换为“修改预订”，继续调整后覆盖今日唯一预订单；次日不带入昨日数量。
- `首页操作层级` 首页只保留三段：套餐额度、今日菜品、底部地址与提交篮子；“今日还可以预订”等解释性文案不再占用首屏空间，避免窄屏文字溢出。
- `无套餐状态` 首页仍可浏览今日菜品，但加菜和提交入口禁用，提示“暂无有效套餐，不能下单”；购买套餐入口和微信支付文案仅做预留，当前不进入真实支付。
- `修改已预订内容` 在截单前开放，进入修改态后使用与首页一致的三列菜品图片网格；不再使用历史版顶部编辑横条，编辑状态合并进套餐次数卡，卡片只强调总次数、已用和剩余，底部地址与已选浮层主按钮改为“保存修改”，确认页展示原订单和本次变化。
- `提交与修改确认` 不使用系统原生确认弹窗，改为 Figma 对应的全屏确认层：摘要卡展示本次预订/修改、重量和额度进度，明细卡展示菜品变化，地址卡展示配送地址和联系人，底部提供“确认提交/保存修改”和“继续调整”。
- `订单页` 与 Figma `05 小程序 / 订单` 对齐，只展示待发货、已发货、已签收、已取消四个状态标签，不展示“全部”和刷新按钮；订单卡保留订单号、菜品摘要、时间、状态 badge 和右下角单按钮。待发货按钮进入取消/修改操作面板，以保留修改已预订内容的真实入口。
- `地址管理` 从“我的-地址管理”进入，页面只承担地址列表、默认地址切换、编辑和删除；新增/编辑使用底部弹框表单，带遮罩和关闭入口，不再像独立页面一样把表单常驻在正文里。地址列表中手机号做脱敏展示，完整号码仅用于表单提交和接口数据。
- `地址管理原型` 已补充为 Figma 节点 `07 小程序 / 地址管理`：使用自定义顶部和返回入口，正文展示地址数量、新增地址按钮、默认地址卡、普通地址卡；卡片操作包括设为默认、编辑、删除；新增/编辑态使用底部弹框表单，字段覆盖收货人、联系电话、所在地区级联选择、详细地址和保存按钮。
- `文字溢出` 已对提交确认和修改预订页做短句处理，底部篮子、提示卡和地址行需要按小程序窄屏宽度预留换行空间。
- `套餐与我的` 参考瑞幸的会员身份卡片感，但内容改为蔬菜套餐权益、剩余额度、今日预订和常用服务；`06B 小程序 / 我的（多套餐摘要）` 只展示最早创建的第一个套餐摘要，避免个人页承载复杂切换；常用服务保留订单、地址管理、套餐、账号设置，协议、隐私和退出登录统一收进“账号设置”入口。`11B 小程序 / 套餐（多套餐切换）` 用左右滑动卡片展示用户所有套餐，顺序按套餐创建时间从早到晚，卡片下方的权益、周期用量和使用明细跟随当前选中套餐切换。

菜品图片来自 Unsplash，本地保存用于 Figma 原型稳定捕获；正式上线前需要替换为自有菜品照片或已授权素材。

Figma MCP 当前不支持 `saveVersionHistoryAsync` 主动保存命名历史版本；每次同步后以 Figma 自动历史和本文件记录为准，若需要命名版本，需要在 Figma 客户端手动保存版本。
