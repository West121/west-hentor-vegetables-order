# 蔬菜预订系统设计说明

日期：2026-06-17

## 目标

开发一套社区蔬菜预订系统，覆盖小程序用户下单、后台运营管理、套餐和库存闭环。现有 PRD 是业务规则依据，3 个 HTML 文件作为交互和信息结构参考，但正式系统需要重新设计视觉和前端结构。

第一版目标是做出可运行、可继续扩展的 MVP：

- PC 管理后台：订单、会员用户、套餐、用户套餐、菜品、任务、发货统计、后台用户与权限。
- 小程序端：微信登录、首页选菜、地址管理、订单、套餐、我的。
- 后端 API：统一服务小程序和后台，保证套餐次数、库存、订单状态的事务一致性。
- 数据库：PostgreSQL + Prisma，优先保证表结构清晰、事务可靠、审计可追踪。

## 后端选型

推荐第一版使用 Next.js 全栈后端，而不是先拆 Spring Boot 或 Go 服务。

### 方案对比

方案 A：Next.js 全栈 + Prisma + PostgreSQL

- 优点：后台页面、API、权限、类型、表结构在一个工程内闭环，适合快速交付 MVP。
- 优点：和 Web 管理端技术栈一致，减少服务拆分、联调和部署复杂度。
- 风险：长期复杂履约、定时任务、外部系统集成变多时，需要拆服务或引入任务队列。
- 结论：推荐。

方案 B：Spring Boot + Next.js 管理端 + Taro 小程序

- 优点：后端分层、事务、权限和企业项目治理成熟。
- 风险：第一版开发量更大，前后端契约和部署复杂度更高。
- 适合：业务已经确定长期多人维护、接口量和后台流程会快速膨胀。

方案 C：Go + Next.js 管理端 + Taro 小程序

- 优点：部署简单，性能好。
- 风险：后台业务 CRUD、权限、数据校验和事务代码会更偏手写，初期效率不如 Next.js 或 Spring Boot。
- 适合：对性能、部署体积、长期服务稳定性有明确优先级。

## 总体架构

采用 monorepo：

- `apps/admin-web`：Next.js 管理后台，React + shadcn/ui + Tailwind。
- `apps/admin-web/app/api`：第一版与后台共用 Next.js runtime，暴露小程序和后台 API。
- `apps/miniapp`：Taro + React + NutUI 微信小程序。
- `packages/db`：Prisma schema、迁移、seed 数据。
- `packages/shared`：枚举、Zod schema、类型、业务常量。
- `packages/ui`：如果后台组件沉淀明显，再抽共享 UI；第一版不强行抽象。

后台与小程序通过统一 API 通信，不让小程序直接接触数据库。

## 用户体系设计

会员用户和后台用户不融合。

原因：

- 会员用户是业务对象，来自微信小程序，身份字段是 `openid`、手机号、昵称、地址、套餐和订单。
- 后台用户是操作身份，负责登录后台、变更订单、调整库存、配置套餐、管理权限。
- 两者审计和权限边界不同。把它们混在 `users` 表中会让登录方式、状态、权限、数据隔离变复杂。

### 会员用户管理

后台新增“会员用户”模块，管理 `users` 业务用户：

- 按手机号、昵称、openid、状态搜索。
- 查看用户基础资料、套餐、地址、订单历史。
- 禁用或启用用户；禁用后小程序可登录但业务操作被锁定并展示原因。
- 给用户绑定套餐、冻结或解冻用户套餐。
- 查看用户套餐消耗记录和操作日志。

会员用户表仍使用 PRD 中的 `users`，必要扩展：

- `disabled_reason`：禁用原因。
- `remark`：后台备注。

### 后台用户与权限

后台新增“后台用户与权限”模块，管理后台登录身份：

- 后台用户新增、编辑、禁用、重置密码。
- 角色配置：超级管理员、运营管理员。
- 权限控制：菜单可见、操作按钮可用、数据范围。
- 操作日志：记录管理员对订单、库存、套餐、任务、用户状态的关键变更。

新增表：

- `admin_users`：后台账号。
- `admin_roles`：后台角色。
- `admin_user_roles`：账号角色关系。
- `admin_permissions`：权限点。
- `admin_role_permissions`：角色权限关系。
- `admin_operation_logs`：后台操作日志。

第一版权限可以先实现角色级权限，不做复杂自定义策略。数据范围先预留 `scope_type` 和 `scope_value`，用于后续按小区或站点隔离。

## PC 管理后台信息架构

后台第一版使用左侧导航 + 顶部工具栏 + 内容区。视觉上不照搬演示 HTML 的大圆角和 emoji 风格，改成清爽、稳定、适合高频操作的运营后台。

左侧导航需要支持展开和折叠：

- 展开态宽度 220px，显示品牌、一级分组和二级菜单；二级菜单字号不低于 15px，分组标题不低于 13px。
- 折叠态宽度 72px，只显示图标入口和当前状态；鼠标悬停时显示一级/二级菜单提示，点击图标进入该分组默认二级页。
- 折叠状态作为后台用户偏好保存，刷新页面后保持上次选择。
- 后台正文、表格、筛选器和按钮文字基础字号不低于 13px，避免高频运营页面出现难读的小字。

后台导航采用两级结构：

1. 工作台
   - 今日待发货、今日订单、总重量、库存预警、截单状态。
   - 快捷入口：批量发货、添加菜品、绑定套餐。

2. 订单管理
   - 订单列表：筛选订单号、手机号、下单时间、状态、菜品；查看详情、发货、修改物流、作废待发货订单、导出。
   - 发货统计：按日期、状态、菜品分类、小区地址筛选，汇总订单数、总重量、菜品明细，支持复制和导出。

3. 会员管理
   - 会员用户：搜索会员，查看资料、地址、套餐、订单；禁用、启用、编辑备注。
   - 用户套餐：给用户绑定套餐，修改已用次数、到期日、下次发货日，冻结、解冻并记录操作日志。

4. 套餐管理
   - 套餐模板：新增、编辑、停用、排序；已绑定模板禁止修改总次数和单次重量上限。
   - 菜品管理：新增、编辑、上架、下架、库存调整、图片上传；库存为 0 自动下架；分类决定加减步长。

5. 任务管理
   - 任务配置：创建限时活动任务，配置生效时间、截单时间、活动标签、关联菜品，支持启停和复制任务。

6. 系统管理
   - 后台用户：后台用户账号、角色、权限。
   - 操作日志：记录订单、库存、套餐、任务、会员状态等关键操作。
   - 系统设置：每日截单时间、客服电话、关于我们、隐私协议链接等基础配置。

## 小程序信息架构

小程序使用 Taro + NutUI。第一版保留底部 Tab，但结构比参考 HTML 更完整。底部 Tab 固定为：首页、我的，不设置单独的预订、订单、套餐菜单；预订能力放在首页，订单和套餐从“我的”入口进入。

1. 首页/预订
   - 套餐选择卡片：展示有效套餐、剩余次数、单次重量上限、冻结状态。
   - 无有效套餐时，首页展示“暂无有效套餐，不能下单”提示；菜品可浏览，但加菜和提交入口禁用，引导联系客服或绑定套餐。
   - 截单提示：未截单显示剩余时间，已截单隐藏提交按钮。
   - 菜品分区：采用左侧分类 + 右侧菜品列表，减少跳转和滚动成本。
   - 重量进度：已选重量、剩余重量、超重拦截。
   - 地址卡片：默认地址回填，无地址引导新增。
   - 提交前确认：展示菜品、重量、套餐、地址。
   - 修改已预订内容：待发货且未截单的订单可进入修改态，预填原订单菜品，保存时覆盖原预订并展示变化明细。

2. 订单（我的二级页）
   - 标签：待发货、已发货、已签收、已取消、全部。
   - 待发货可取消，取消需选择原因。
   - 已发货可复制运单号。
   - 已取消可前端删除隐藏。

3. 地址
   - 新增、编辑、删除、设为默认。
   - 单用户最多 10 条。
   - 手机号、省市区、详细地址校验。

4. 我的
   - 微信头像、昵称、手机号。
   - 当前套餐卡、今日预订状态、可修改入口和套餐消耗记录。
   - 常用入口保留“订单”和“套餐”，订单查看历史预订，套餐查看额度、剩余次数和消耗记录。
   - 客服、协议、账号注销入口。
   - 账号禁用时展示禁用原因并锁定业务操作。

5. 套餐（我的二级页）
   - 展示当前套餐名称、单次重量额度、剩余次数、到期时间和下次可预订日期。
   - 展示核心权益：额度、剩余次数、截单前可修改、默认地址自动带入。
   - 套餐页只承载套餐权益和状态，不做复杂营销券体系。

## 核心业务规则

下单事务：

1. 检查用户状态、截单时间、地址归属。
2. 锁定用户套餐，确认有效、未冻结、未用完、未过期；没有有效套餐时直接返回 `PACKAGE_REQUIRED`，前端不允许提交。
3. 锁定菜品库存，确认上架且库存足够。
4. 校验选择步长和总重量不超过套餐上限。
5. 扣减库存，库存为 0 自动下架。
6. 用户套餐已用次数 + 1，更新最近发货日和下次发货日。
7. 创建订单和订单明细，保存地址、菜品名称、单位重量快照。

修改预订事务：

1. 仅待发货且未截单订单可修改。
2. 锁定原订单、用户套餐和涉及菜品库存。
3. 按新明细重新校验套餐额度、菜品步长、库存和地址归属。
4. 按差额返还或扣减菜品库存，库存为 0 自动下架。
5. 覆盖订单明细和重量快照，更新 `orders.modified_at`。
6. 写入 `order_change_logs`，保存修改前后菜品、重量、地址和操作来源。

取消事务：

1. 仅待发货订单可取消。
2. 订单状态改为已取消，记录取消时间和原因。
3. 用户套餐已用次数 - 1。
4. 菜品库存返还。
5. 库存恢复后不自动上架，由运营手动处理。

后台发货：

1. 运营填写运单号后订单变为已发货。
2. 自动写入发货时间。
3. 超级管理员可修正状态，所有修正写操作日志。

## 视觉与交互方向

PC 后台：

- 风格：干净、克制、运营效率优先。
- 颜色：以绿色作为品牌强调色，背景和表格使用中性灰白，不使用大面积绿色渐变。
- 组件：shadcn/ui 的 Table、Dialog、Sheet、Tabs、Form、Select、Badge、Toast。
- 信息密度：列表页优先表格和筛选器，详情、编辑、新建统一使用分区弹窗承载，不跳独立详情页，也不使用右侧抽屉作为主要操作容器。
- 字号：后台列表和表单默认使用 14px 正文，次要辅助信息最低 13px；小程序正文默认 14px，库存、时间等辅助标签最低 12px。
- 后台工作弹窗：订单、会员、套餐、菜品、任务、账号权限等模块的详情、编辑、新建都使用同一个 ModalShell 规范，支持标题栏拖拽、右上角全屏、右下角伸缩；全屏和伸缩不丢失表单状态，未保存关闭需要二次确认。
- 图标：使用 lucide-react，不使用 emoji 作为正式图标。

小程序：

- 风格：轻量、亲切、清晰。
- 首页核心是“套餐剩余 + 默认地址 + 菜品选择 + 底部篮子 + 提交/保存修改”。
- 底部导航只有：首页、我的；预订、订单、套餐都不是独立 Tab。
- 预订页参考餐饮点单页的信息效率：左侧分类固定，右侧菜品列表连续操作，底部固定已选篮子；但视觉保持蔬菜生鲜主题，不做咖啡促销式强营销。
- 我的页和套餐页参考会员产品的身份卡片感，突出当前套餐、剩余额度、今日预订和常用服务。
- 菜品卡片要展示图片、名称、库存状态、步长和已选重量。
- 交互尽量减少跳转，提交前使用确认弹层。

## 原型设计阶段

工程实现前必须先完成原型设计确认。当前 PRD 和 HTML 参考只能作为业务和信息结构输入，不能直接作为正式视觉稿。

### 视觉方向探索

先产出 3 个差异明确的视觉方向供选择：

1. 高效运营型：PC 后台信息密度更高，表格、筛选、批量操作优先；小程序保持清爽轻量。
2. 社区生鲜型：小程序更有生活感和商品感，后台仍保持克制；适合强调用户体验和品牌识别。
3. 履约看板型：后台突出待发货、库存预警、今日重量和任务状态；适合运营每天高频处理订单。

每个方向至少覆盖：

- PC 后台工作台或订单管理首屏。
- 小程序首页预订首屏。
- 颜色、字体、圆角、图标、组件密度和信息层级说明。

用户选择方向后，再进入具体原型。

### 低保真流程原型

低保真原型用于确认流程，不追求视觉细节：

- 小程序：首页选菜 -> 提交确认 -> 修改已预订内容 -> 我的 -> 订单 / 套餐 / 地址管理。
- PC 后台：工作台 -> 订单管理 -> 会员详情 -> 菜品库存 -> 用户套餐绑定。
- 重点验证：导航结构、字段完整性、操作入口、状态变化、异常提示。

### 高保真交互原型

高保真原型用于指导开发：

- PC 后台至少包含：工作台、订单管理、会员用户、菜品管理、用户套餐、后台用户与权限。
- 小程序至少包含：首页预订、首页无套餐、提交与修改确认、修改已预订内容、订单、地址管理、套餐、我的。
- 使用真实业务字段和模拟数据。
- 关键交互要可点击：筛选、后台详情/编辑/新建弹窗、弹窗拖拽/全屏/伸缩状态、状态切换、加减菜品、提交确认、截单前修改预订。

### Figma 与本地原型

优先产出 Figma 设计稿，便于后续评审和标注。如果没有现成 Figma 文件，则新建 Figma 文件或先用本地可点击原型承载高保真设计。

本地可点击原型只用于快速验证交互；最终工程实现仍按 Next.js + Taro 技术栈重建，不把一次性原型代码直接当生产代码。

## 错误处理

- 表单错误在字段附近显示。
- 业务错误用 toast 或 dialog，但库存不足、套餐不足、已截单需要明确原因。
- 后台批量操作返回成功数、失败数和失败原因。
- API 返回统一结构：`success`、`data`、`error.code`、`error.message`。

## 测试策略

单元测试：

- 重量计算、步长校验、订单状态流转、权限判断。

集成测试：

- 取消订单返还库存和套餐次数。
- 库存为 0 自动下架。
- 管理员权限拦截。

前端验证：

- 管理后台主要表单和列表操作。
- 小程序首页选择重量、超限拦截、无地址引导、截单状态。

## 第一版实施顺序

1. 完成 3 个视觉方向探索，并选择一个方向。
2. 产出 PC 后台和小程序低保真流程原型。
3. 产出 PC 后台和小程序高保真交互原型，并确认核心页面。
4. 创建 monorepo、Next.js 后台、Taro 小程序、Prisma 基础包。
5. 建立数据库 schema、迁移和 seed 数据。
6. 实现后台登录、后台用户、角色和权限基础。
7. 实现订单、菜品、套餐、会员用户核心 API。
8. 按已确认原型实现 PC 后台核心页面。
9. 按已确认原型实现小程序预订链路和我的页面。
10. 补充统计、导出、操作日志和关键测试。

## 明确不做进第一版的内容

- 套餐赠送或分享管理。
- 复杂售后流程。
- 物流公司接口直连。
- 多仓、多小区复杂数据权限。
- 自动签收定时任务。
- 多租户 SaaS 化。

这些能力可以在核心闭环跑通后继续扩展。
