# 后台品牌、折叠菜单与账号校验调整设计

## 背景与目标

本次调整覆盖管理后台品牌、门店显示名、折叠侧栏交互、后台账号校验和面单打印菜单文案，目标如下：

- 后台系统名称从 `Hentor Fresh` 调整为 `HanYang Fresh`，并支持在系统设置中修改。
- 后台系统名称全平台统一，不随当前门店切换而变化。
- 将现有“莲花分店”相关门店显示名统一改为“涵养总店”。
- 侧边栏折叠后，鼠标从一级图标移动到二级菜单时，菜单保持显示并允许点击跳转。
- 新建后台用户时，登录账号包含任何空白字符都必须被拒绝。
- 管理后台菜单“快递100打印机”改为“面单打印”，底层快递100能力和接口名称保持不变。
- 后台登录页增加微信扫码登录；首次扫码完成原后台账号绑定，之后可直接扫码登录。

## 方案选择

项目现有系统配置以门店为作用域，存储在 `system_config` 中。为了避免为单个字段引入新的全局配置表，本次新增 `admin_system_name` 配置项，并采用“写入时同步所有门店、读取时统一回退”的方式实现全平台一致：

- 任意门店的系统设置保存 `admin_system_name` 时，将同一值同步到所有门店。
- 读取当前门店配置时，优先读取 `admin_system_name`；缺失或为空时使用 `HanYang Fresh`。
- 新增门店尚未生成该配置时，也可通过默认值正常显示；下一次保存系统设置时完成同步。

该方案复用现有表结构、接口和操作日志，迁移风险小，同时满足后台可配置和全平台统一的要求。

## 详细设计

### 1. 后台系统名称

系统设置表单新增“后台系统名称”字段：

- 默认值：`HanYang Fresh`。
- 必填，保存前去除首尾空白。
- 后端拒绝空值，避免后台标题显示为空。
- 保存成功后，系统设置接口返回最新名称。

管理后台数据加载完成后，将该名称传给 `AdminShell`，覆盖垂直、双栏和横向三种布局中的品牌展示。若接口不可用或当前没有可读取的门店配置，则使用 `HanYang Fresh`。

页面静态元数据、favicon 可访问文本和初始化数据中的后台旧名称一并改为 `HanYang Fresh`。小程序的 `login_title` 仍是独立配置，不与后台系统名称强制绑定。

### 2. 门店名称

门店主数据 `seed-store-lotus` 的显示名改为“涵养总店”，覆盖：

- Spring Boot 初始化数据。
- Prisma 种子数据。
- MySQL 全量初始化脚本。
- 一条可重复执行的增量 SQL，用于更新测试环境和已有数据库。

增量 SQL 按稳定门店 ID `seed-store-lotus` 更新，避免误改用户地址、配送说明或测试样例中的“莲花小区”地理文本。

### 3. 折叠侧栏二级菜单

当前二级菜单通过 `margin-left` 与图标分隔，图标和菜单之间存在不属于悬停区域的空隙，鼠标经过空隙时会触发菜单隐藏。

调整后：

- 使用包含透明内边距的二级菜单外层作为连续悬停桥接区，不再使用脱离命中区域的外边距。
- 图标、桥接区和菜单卡片都属于同一个悬停容器。
- 增加 `focus-within` 展示规则，键盘聚焦菜单项时菜单保持显示。
- 二级菜单链接继续使用现有 `section` 查询参数跳转，点击行为不变。
- 不增加长时间强制驻留，鼠标真正离开图标与二级菜单整体区域后正常收起。

### 4. 后台登录账号空格校验

校验仅作用于“新建后台用户”的登录账号字段，不改变已有账号，也不自动删除用户输入中的内部空白。

三层校验保持一致：

1. 前端表单：提交前检查任意空白字符，立即展示字段错误。
2. Spring Boot 服务：创建用户的账号标准化逻辑拒绝任意 Unicode 空白字符。
3. TypeScript 业务层：保留与 Spring Boot 一致的兜底规则，避免不同运行路径行为不一致。

统一提示文案：`当前账号存在空格，请重新填写`。

空账号仍提示“请输入登录账号”；已存在账号仍提示“登录账号已存在”。校验顺序为必填、空白字符、唯一性。

### 5. 面单打印菜单文案

以下用户可见入口改为“面单打印”：

- 系统管理侧栏二级菜单。
- 首页系统就绪状态卡片中的对应入口名称。

组件名、路由 section `kuaidi-printers`、API 路径、数据库表和快递100供应商相关说明保持不变，避免无关重构。

### 6. 微信扫码登录

后台微信登录采用微信开放平台“网站应用”OAuth 2.0 标准流程，使用独立的网站应用凭证，不能复用现有微信小程序凭证。

#### 6.1 登录流程

1. 登录页保留账号密码登录，同时增加“微信扫码登录”入口。
2. 用户点击入口后，后端生成一次性 `state`，保存短时授权上下文，然后跳转到微信官方网站应用授权地址。
3. PC 浏览器由微信官方页面展示二维码；用户扫码并确认后，微信携带 `code` 和 `state` 回调后台。
4. 后端校验 `state`，使用 `code` 换取网站应用微信身份，得到 `openid`，并在微信返回时记录 `unionid`。
5. 已绑定后台用户：直接签发项目现有后台会话 Cookie，跳转运营工作台。
6. 未绑定后台用户：签发短时、一次性的绑定凭证，回到登录页进入“绑定后台账号”状态。
7. 用户输入原后台登录账号和密码；验证账号存在、状态正常且密码正确后，将微信身份绑定到该后台用户，再签发正常后台会话。

微信授权不会自动创建后台管理员，也不会根据昵称或手机号猜测账号，现有角色、权限和门店数据范围完全沿用被绑定的后台账号。

#### 6.2 微信身份绑定

新增独立的后台微信绑定数据结构，至少包含：

- 后台用户 ID，唯一，保证一个后台用户只绑定一个网站应用微信身份。
- 网站应用 `openid`，唯一，保证一个微信身份不能绑定多个后台账号。
- `unionid`，微信返回时保存，用于同一开放平台主体下的身份关联，但不作为本次登录的唯一依赖。
- 创建时间、更新时间和最近微信登录时间。

首次绑定必须再次验证原后台账号密码。已停用账号即使微信身份已绑定也不能登录。后台账号删除时同步删除绑定；后台账号被停用或会话被踢下线时，微信登录不能绕过现有状态与会话管理。

#### 6.3 本机微信快速登录边界

普通浏览器页面不探测本机微信进程，也不读取桌面微信登录状态。点击“微信扫码登录”后统一进入微信官方授权流程：

- 微信官方页面若针对当前 PC 微信版本和环境提供“使用已登录微信快速确认”或客户端拉起能力，由微信官方完成。
- 微信官方未提供快速确认时，仍展示标准二维码扫码登录。
- 产品文案只承诺“微信扫码登录”，不承诺所有环境都能自动识别本机微信或免扫码。

#### 6.4 配置与接口

新增独立环境变量：

- `WECHAT_OPEN_APP_ID`：微信开放平台网站应用 AppID。
- `WECHAT_OPEN_APP_SECRET`：微信开放平台网站应用 AppSecret，仅后端读取。
- `WECHAT_OPEN_REDIRECT_URI`：公网 HTTPS 回调地址，域名必须与微信开放平台授权回调域一致。

登录页通过不包含密钥的公开状态接口判断微信登录是否已配置。未配置完整时隐藏微信登录入口，并保留账号密码登录，不影响原有后台使用。

需要新增微信授权发起、微信回调、首次绑定三个后端入口。所有微信密钥交换都在 Spring Boot 后端完成，前端不得接触 `AppSecret` 或微信访问令牌。

#### 6.5 安全要求

- `state` 使用强随机值、短时有效、单次消费，防止登录 CSRF 和回调重放。
- 首次绑定凭证短时有效、单次消费，并只允许绑定本次微信回调获得的身份。
- 微信 `code` 只使用一次，错误响应不输出 `AppSecret`、访问令牌或完整微信响应。
- 回调只允许固定站内跳转目标，禁止使用未经校验的任意 `redirect` 参数。
- 账号密码校验复用现有密码编码器和账号状态检查，不复制弱化版认证逻辑。
- 成功绑定与微信登录写入操作日志，记录后台用户和结果，不记录密钥、令牌或密码。

## 数据与兼容性

- `system_config` 不新增表结构，只新增配置键 `admin_system_name`。
- 新增后台微信身份绑定表及对应索引，不改会员小程序用户的微信身份字段。
- 全量初始化数据写入 `HanYang Fresh`。
- 增量 SQL 同时更新已有门店名称，并补齐或更新后台系统名称配置。
- 现有 `login_title` 继续服务小程序登录页，不受本次后台品牌调整影响。
- 历史账号中即使包含空白也不做自动迁移，避免产生账号冲突或无法预期的登录变化；新建账号开始执行新规则。
- 微信网站应用与小程序使用不同凭证和 `openid`；若两者绑定到同一微信开放平台主体，可保存 `unionid`，但后台权限仍只认后台微信绑定关系。

## 错误处理

- 后台系统名称为空：前端阻止提交，后端返回明确的参数错误。
- 配置缺失：后台显示默认值 `HanYang Fresh`。
- 账号包含空白：前端字段提示与后端错误消息一致。
- 增量 SQL 重复执行：使用稳定 ID 更新和 upsert 方式，结果保持一致。
- 微信配置缺失：保留账号密码登录，微信登录入口不进入不可完成的授权流程。
- 微信授权取消、超时或 `state` 无效：回到登录页显示明确错误，不创建会话或绑定。
- 微信身份未绑定：进入一次性账号密码绑定流程，不自动创建后台账号。
- 微信身份已绑定但后台账号停用：拒绝登录并提示联系管理员。

## 测试与验收

### 自动化验证

- 系统设置表单 payload、必填校验和默认值测试。
- Spring 系统设置读写、全平台同步和缺失回退测试。
- 前端后台用户表单空白字符校验测试。
- Spring 与 TypeScript 创建后台用户空白字符拒绝测试。
- 导航配置和首页卡片文案测试。
- 折叠菜单源码约束或组件行为测试，确认连续悬停区域和 `focus-within` 存在。
- 微信登录配置状态、授权 URL、`state` 单次校验和回调错误处理测试。
- 首次扫码绑定、已绑定扫码直登、重复绑定拒绝和停用账号拒绝测试。
- 验证微信密钥、访问令牌和账号密码不会写入日志或返回前端。
- 管理端测试、构建以及 Spring 测试、编译。

### 浏览器验收

1. 后台侧边栏显示 `HanYang Fresh`。
2. 系统设置可修改后台系统名称，刷新后名称保持一致。
3. 切换门店后后台系统名称不变化。
4. 折叠侧栏后，从图标移动到二级菜单不会消失，菜单项可点击并完成跳转。
5. 新建后台用户输入半角空格、全角空格或制表符时，显示“当前账号存在空格，请重新填写”。
6. 系统管理菜单和首页入口显示“面单打印”。
7. 门店选择及相关后台页面显示“涵养总店”。
8. 未配置微信网站应用时，账号密码登录正常，微信登录入口不显示。
9. 首次微信扫码后需输入原后台账号密码，绑定成功后进入工作台。
10. 退出后再次使用同一微信扫码，无需再次输入后台账号密码即可登录。
11. 使用未绑定微信、已绑定微信、停用后台账号分别得到符合设计的结果。

### 微信开放平台实测前置条件

真实微信扫码验收需要用户提供或完成以下准备：

- 已认证并审核通过的微信开放平台“网站应用”。
- 网站应用 `AppID`。
- 网站应用 `AppSecret`，通过测试服务器环境变量或安全凭据方式配置，不在聊天、代码、文档或前端中明文传递。
- 已备案并可通过公网 HTTPS 访问的测试域名。
- 在微信开放平台中配置授权回调域，并确认与 `WECHAT_OPEN_REDIRECT_URI` 的域名完全一致。

在上述真实凭据尚未准备完成时，可以完成代码、自动化测试和模拟回调验证，但不能声明真实微信扫码链路验收通过。

### 测试环境交付

自动化和浏览器验收通过后，部署管理端前端与 Spring Boot 后端到测试环境，并执行增量 SQL 更新现有门店、后台系统名称配置和微信绑定数据结构。部署后重新验证后台名称、门店名称、折叠菜单、面单打印入口以及微信扫码与首次绑定流程。
